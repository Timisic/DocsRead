## Part 1 用最少的概念讲清楚 OpenClaw 是啥

### 1.1 一句话的“心智模型”

OpenClaw 不是一个只会聊天的 bot，它更像一个“带工具权限的 AI 助手”，通过 gateway 把能力接到不同平台上。

### 1.2 记住 3 个词就够了：Gateway / Workspace / Skill

[提示：这里翻到“术语速查表"]

刚开始接触 OpenClaw，其实不用一下子记很多概念。先记住 3 个词就够了：Gateway、Workspace、Skill。

- **Gateway**：后台网关进程，负责“收消息、调度任务”。它活着，你的 bot 才可能活着。
- **Workspace**：助手的“人格 + 记忆 + 行为边界”的存放地。比如 SOUL.md、USER.md 这些。
- **Skill**：可复用能力模块。论文推送、自动摘要、定时任务、远程 coding，本质都是 skill + 配置。

### 1.3 现实提醒：它强但也“很能折腾”

OpenClaw 的优点是高度可配置。反过来说，它的门槛也正是在这里。
你越想把它用顺，前期越要把配置习惯立住。
所以我今天会反复强调一句话：**先查 docs，再改配置；先看 diff，再写入；做完能验证，出事能回滚，所以要做好git备份。**

## Part 3 Workspace：你不需要背配置，但要知道“边界在哪里”

### 3.1 Workspace 和配置目录的区别（用一句话讲明白）

OpenClaw 有两个“家”：

- `~/.openclaw/` 更像系统配置、凭证、会话
- `~/.openclaw/workspace/` 是助手的“人格、规则、偏好、记忆”

路径不一定要背，但你至少要知道：**Workspace 决定的是它怎么做事，以及它做事的边界。**

[提示：翻到 “SOUL.md 配置 / 什么是 Workspace” 那一段]

### 3.2 你只要记住：哪几个文件“每次都会加载”

[可以打开我mac本地 soul.md等]

每次会话开始时，系统都会加载几个核心文件：
SOUL.md、USER.md、IDENTITY.md、AGENTS.md、TOOLS.md。
所以你要让它长期遵守的规则，最好写在这些文件里，而不是只在聊天里临时说一句。

### 3.3 安全提醒：Workspace 里不要放敏感信息

这个很重要：
API Key、Token、密码这种东西，不要写进 Workspace。
你可以让它记“偏好”和“流程”，但不要让它记“秘密”。

[提示：这里停顿一下，强调“不要把 token 贴进群聊/不要写进 workspace”】

### 3.4 你跟 AI 说“怎么改 Workspace”才靠谱

口令尽量固定成这种风格：
“请更新 USER.md，新增我的偏好：回答先给结论；改配置前先给 diff；高风险操作要二次确认。”

## Part 4 接入聊天平台：只讲关键点 + 让它在聊天里“自我解释”

### 4.1 Discord/飞书接入：你只需要记 3 个关键点

接聊天平台这块最容易卡，但你先记住 3 个最容易出问题的点就够了：

1. **Token 写对地方**（`~/.openclaw/openclaw.json` 那类配置里）
2. Discord 的 **Message Content Intent** 要开（不然它读不到消息内容）
3. 配置改了要 **重启 gateway**（`openclaw gateway restart`）

至于 Developer Portal 里每一步怎么点，这些细节文档里都有，我今天就不在这儿展开。

### 5.2 场景二：远程代码编写（这块最值得讲“安全闭环”）

远程写代码这件事听起来很爽，但最怕的是：它改错文件、或者直接 push。
所以文档里给的那套“安全基线”，我非常建议你把它固定成习惯：

- 限定操作目录（比如只允许 `~/projects/`）
- 改文件前先给计划和 diff，你确认后再写入
- git push / 删除 / 覆盖配置 必须二次确认
- 默认不要直接推送

只要这个流程立住了，它就不是“危险的远程控制”，而是“可控的远程协作”。

### 5.4 配置管理：一句话解决“改崩了怎么办”

更新也好，让 OpenClaw 帮你改配置也好，都有可能改崩。
所以最简单的办法就是：**尽量用 Git 管理 `~/.openclaw`。**

每次改完配置，确认能用之后就 commit 一次。真改崩了，优先用 `git revert` 回滚。

我自己的做法是：
网关起不来就先 `openclaw doctor`；
最近刚改过配置如果实在改不回来就先 revert；
恢复后再 `gateway status` + 聊天里发一句“你好”验证链路。

## Part 6 最后补一句：安全、风险，以及你到底需不需要 OpenClaw

### 6.1 我想把最现实的一句话放在最后

OpenClaw 很强，这个没问题。
但它不是那种“装上就一定提升效率”的工具，更不是所有人都一定有必要用。

如果你没有一个很明确、很直接、而且会反复发生的需求，那大多数时候，你其实没必要上 OpenClaw。
因为它的前期配置、后期维护、排错成本，是真实存在的。

### 6.2 它的风险不是“回答错了”，而是“可能动错手”

这一点我建议一定要讲清楚。
普通聊天型 AI，很多时候答错了也就是答错了。
但 OpenClaw 不一样，它一旦接了工具、接了文件系统、接了远程操作能力，风险就会变成真的操作风险。

如果你边界没设好，它不只是会改错配置。
严重一点，它可能误删你项目里的重要文件；再严重一点，甚至可能碰到系统相关目录，把你的环境搞坏。

所以它最需要防的，不是“它聪不聪明”，而是“它有没有在你允许的边界内做事”。



### 收尾：给一个能带走的“固定话术”

最后我还是想把一句最实用的话留给大家。

**先查 docs，再给最小方案；先看 diff，再允许写入；做完要验证，平时用 Git 备份，出事就能回滚。**

把这个流程坚持下来，OpenClaw 就会越用越顺，而不是越改越乱。
