# OpenClaw 文档反馈（新手视角，2026-02-28）

## 评估前提

假设我是以下读者：

- 之前没接触过 OpenClaw
- 有一点代码基础（会用终端，会看配置文件）
- 用过 AI 产品，但不熟悉 Agent/Skill/Gateway 体系

---

## 一句话结论

这 9 篇文档已经具备“可上手”的完整链路，尤其 05/06 的实操性明显提升；但对新手来说，仍有几个地方容易“看懂文字，但不确定自己是否做对了”。

---

## 我作为新手最喜欢的点

1. 阅读路径完整：`index -> 安装 -> 配置 -> 场景 -> 维护/FAQ` 很顺。
2. 每章“本章产出”很有帮助，知道为什么要读这一章。
3. 05/06 给了可直接复制的对话模板，降低了起步门槛。
4. 风险意识更强了：Token 安全、先 diff 后执行、push 前确认，这些都很关键。

---

## 我作为新手仍会卡住的点（按优先级）

### P0（建议优先补）

1. **缺少“全局术语速查”**  
   新手会反复遇到 `workspace / gateway / onboarding / skill / guild / intents`。  
   建议加一页或在 `index` 增加术语表（每个术语一句话 + 文件路径）。

2. **“成功标准”还可以更可视化**  
   现在很多地方是文字描述“应该成功”。新手更需要“成功时会看到什么原文输出”。  
   建议关键节点加“预期输出样例框”（例如 gateway status、Discord 回复截图文字样例）。

3. **配置文件版本差异提醒还不够醒目**  
   你已经写了“以官方文档为准”，但新手容易忽略。  
   建议在 `01`、`03`、`08` 顶部再加一条同样提示，并使用统一警示样式（如 `> 注意`）。

### P1（建议第二批补）


5. **FAQ 可以更“故障树”**  
   现在 FAQ 可读性不错，但新手更习惯“如果 A 现象 -> 先做 B -> 再做 C”。  
   建议把 Q1/Q3 改成决策树样式（文本版即可）。

6. **缺少“第一次完整演练”入口**  
   新手最需要一个 15 分钟“从 0 到消息回执”的最短路径。  
   建议在 `index` 增加“快速演练：只做 6 步，10-15 分钟跑通”。

### P2（锦上添花）

7. **对 AI 说的话与终端命令已区分，但可再统一格式**  
   建议统一两个固定小标题：`终端命令`、`对 AI 说的话`，每章保持一致。

8. **可以补“高频误区”小节**  
   例如：  
   - 把 Token 发到群里  
   - 直接让 AI 改配置但不看 diff  
   - 远程修复时直接 push 到主分支

## 分文档反馈（逐篇）


## `01-installation.md`

- 优点：前置条件与排错更实用。
- 建议：在每一步后补“成功信号”1 条（例如命令返回关键字段）。

## `02-soul-config.md`

- 优点：文件职责和加载时机讲清楚了。
- 建议：再加一个“最小可用 USER.md 示例”，让新手立刻照抄可用模板。


## `04-core-principles.md`

- 优点：提示词模板非常实用。
- 建议：可加“反例 -> 改写后”的 1-2 个真实配置案例（更贴近 05/06）。

## `05-scenario-news.md`

- 优点：结构清晰，验证清单实用，去掉了误导性的“按引用量排序”。
- 建议：补 1 个失败案例（比如“没推送到频道”时如何排查）。

## `06-scenario-coding.md`

- 优点：安全流程明显更稳，尤其“先 diff 再执行”。
- 建议：补一句“默认不在生产主分支直接 push”，强化风险边界。

## `08-config-management.md`

- 优点：从 `checkout` 改为 `revert` 对新手非常友好。
- 建议：加“什么时候用 revert，什么时候用新分支修复”的一句话判断。

## `09-faq.md`

- 优点：去掉无效信息后更简洁。
- 建议：Q3 里加“恢复后验证步骤”（例如先 `gateway status` 再发送测试消息）。
