# 核心原则：查阅 Docs

这是使用 OpenClaw 最重要的原则，必须牢记。

---

## 本章产出

完成本章后，你应该已经：

- 知道为什么配置前必须先查官方文档
- 学会给 AI 下“可执行、可验证”的配置指令
- 能避开最常见的三类配置错误

---

## 为什么必须查阅 Docs？

OpenClaw 是一个非常灵活的系统，你可以修改它的提示词、配置、Skills 等几乎所有东西。这种高自由度是一把双刃剑。

### 好处

- 可以完全定制你需要的功能
- 能适应各种特殊场景

### 风险

- 不当的配置可能导致系统崩溃
- 配置越改越复杂，最后难以维护
- AI 可能产生幻觉，生成不存在的配置项

---

## 如何正确配置

### 错误示范（不要这样说）

```
"帮我配置一个流式输出功能"
"帮我优化一下配置"
"帮我添加一个新功能"
```

### 正确示范（建议模板）

```
"请查阅 /docs/ 下关于流式输出的文档，然后按照文档说明帮我配置流式输出功能"
"请查阅 /docs/config/ 下关于性能优化的文档，告诉我有哪些可以优化的配置项"
"请查阅 /docs/features/ 下关于 XXX 的文档，帮我实现这个功能"
```

还可以用这个通用模板：

```text
请先查阅 /docs/<path>/ 下关于 <功能名> 的文档，
再给我：
1) 需要修改的文件和字段
2) 最小变更方案
3) 修改后的验证步骤
4) 回滚方案
```

---

## 配置前的检查清单

在配置任何功能之前，请确保：

- [ ] 告诉 OpenClaw 查阅相关 Docs
- [ ] 确认 AI 理解了文档内容
- [ ] 询问配置的具体步骤
- [ ] 逐步执行，每一步都确认成功
- [ ] 测试配置是否正常工作

---

## OpenClaw 文档结构

OpenClaw 的官方文档位于 `/docs/` 目录下，主要包含：

| 目录 | 内容 |
|-----|------|
| `/docs/channels/` | 各聊天平台的配置文档 |
| `/docs/skills/` | Skills 的使用和配置文档 |
| `/docs/config/` | 配置项的详细说明 |
| `/docs/features/` | 各功能的实现方法 |
| `/docs/api/` | API 接口文档 |

## 常见错误及避免方法

### 错误 1：直接让 AI 配置

```
"帮我配置一个 Cron 任务"
```

**正确做法**：
```
"请查阅 /docs/ 下关于 Cron 任务的文档，然后帮我配置一个每天早上 8 点执行的任务"
```

### 错误 2：跳过文档直接修改

```
"直接修改配置文件，添加这个参数"
```

**正确做法**：
```
"请查阅配置文档，确认这个参数的正确写法和位置，然后再修改"
```

### 错误 3：一次性修改太多

```
"帮我优化所有配置"
```

**正确做法**：
```
"请查阅文档，先帮我优化这一个配置项，成功后再做其他的"
```

## 真实场景改写示例

### 示例 1：论文推送配置

反例：

```text
帮我把论文推送弄好
```

改写后：

```text
请查阅 /docs/skills/ 中与 arxiv 推送相关的文档，
然后按最小可用方案帮我配置：
- 分类：cs.AI, cs.LG
- 每天 08:30 推送到 Discord #paper-daily
修改前先给我 diff，修改后给验证步骤。
```

### 示例 2：远程修复代码

反例：

```text
你直接修好然后 push
```

改写后：

```text
请先查阅 /docs/skills/ 里 coding 相关文档，
再按以下流程处理：
1) 定位问题并给修复计划
2) 先展示 diff，不直接提交
3) 我确认后再应用改动并跑测试
4) 测试通过后再次确认是否允许 push
```
